# ТЗ-ассистент ABAP

ИИ-помощник для SAP-консультантов: помогает писать и дорабатывать технические
задания (ТЗ) на разработку для ABAP-программистов.

- **Backend** — Python: FastAPI + LangGraph (агент с инструментом `update_spec`,
  состояние сессий — в SQLite-чекпоинтере LangGraph, переживает перезапуск).
- **Frontend** — React (Vite + TypeScript): чат, кнопка 📎 для загрузки
  существующего ТЗ (.docx / .md / .txt — основной сценарий), живое
  Markdown-превью документа ТЗ, боковая панель с историей сессий (как в
  ChatGPT) — можно вернуться к работе над ТЗ позже.
- **LLM** — любой OpenAI-совместимый шлюз (настраивается через `.env`).
- **RAG** — Qdrant с базой примеров реальных ТЗ: агент получает целые разделы
  похожих документов как образец стиля и оформления — автоматически по запросу
  пользователя и прицельно через инструмент `find_style_examples`.

## Как это работает

Граф LangGraph: `retrieve → assistant → (есть tool-call?) → tools → assistant → END`.

### Документ ТЗ — список разделов, а не одна строка

Документ ТЗ (`app/spec_doc.py`) хранится в состоянии графа как список
адресуемых разделов (`id`, заголовок, уровень, текст), а не как одна большая
строка Markdown. Это принципиально для документов на десятки-сотни тысяч
слов: они физически не помещаются в контекст LLM целиком и не могут быть
перегенерированы моделью заново на каждую правку (лимит выходных токенов).

Загруженный `.docx`/`.md`/`.txt` сначала конвертируется в Markdown, затем
режется на разделы в три уровня, каждый — подстраховка на случай, если
предыдущий не найдёт структуру:

1. Markdown-заголовки `#`..`######` (в `.docx` — стили Word `Heading N`).
2. Эвристики по тексту абзацев — нумерация верхнего уровня («1. …»), абзац
   целиком жирным (включая ручное выделение без стиля Word), строка КАПСОМ.
3. Фиксированное окно по границам абзацев — если в документе вообще нет
   уловимой структуры («идёт сплошняком»).

Раздел, который всё равно оказался больше `SPEC_MAX_SECTION_CHARS`, принудительно
дробится дальше — так один нераспознанный заголовок на весь документ не
превращается в «один раздел на 200 000 слов». Разделы, полученные не по
настоящим Markdown-заголовкам, помечаются как `synthetic` и в оглавлении
выглядят как «⚠ структура не распознана автоматически».

Узел `retrieve` на каждом ходу:

- эмбеддит последний запрос пользователя и ищет в Qdrant разделы реальных ТЗ —
  они попадают в промпт как образцы стиля (без копирования содержимого).
  Кандидаты берутся с запасом, затем порог по score (`RAG_MIN_SCORE`) и дедуп
  по документу (`RAG_MAX_PER_DOC`): примеры из разных ТЗ ценнее нескольких
  кусков одного. Единица индексации — раздел (H1/H2, до 8000 символов),
  эмбеддится заголовок + начало (лимит FRIDA/e5 — 512 токенов). Агент может
  дополнительно запросить примеры под конкретный раздел инструментом
  `find_style_examples`;
- отдельно ищет топ-K релевантных разделов **самого рабочего документа**
  (`app/rag/doc_retriever.py`) — их полный текст, а не документ целиком,
  идёт в промпт вместе с оглавлением (`<toc>`) всех разделов. Эмбеддинги
  разделов кэшируются во внешнем SQLite (`SECTION_INDEX_DB_PATH`) и
  пересчитываются только для новых/изменившихся разделов.

Если Qdrant или сервис эмбеддингов недоступны, ассистент продолжает работать
без примеров/ретривала (предупреждение в логе, в промпт попадают первые
разделы документа без ранжирования).

Узел `tools` исполняет вызовы инструментов ассистента:

- `list_sections` / `get_sections` — прочитать оглавление или полный текст
  нескольких разделов сразу (когда их нет среди уже подгруженных ретривалом);
  объём одного ответа ограничен `SPEC_READ_MAX_CHARS`, лимит шагов графа на
  один ход — `GRAPH_RECURSION_LIMIT`;
- `find_in_document` — точный текстовый поиск по документу (id разделов +
  сниппеты): для идентификаторов вроде VBAK/ME21N он надёжнее и дешевле
  эмбеддинг-ретривала;
- `update_section` / `insert_section` / `delete_section` — правка на уровне
  одного раздела вместо переписывания всего документа;
- `patch_section` — string-replace внутри раздела для мелких правок: LLM не
  перегенерирует раздел целиком ради исправления одного числа;
- инструменты **MCP-сервера ADT tools** (чтение объектов SAP: структуры
  таблиц, программы, поиск объектов). Подключаются на старте из
  `MCP_SERVER_URL` (streamable_http, Bearer-токен из `MCP_AUTH_TOKEN`);
  пишущие инструменты (`create_/update_/delete_/activate_/transport_...`)
  отфильтровываются — ассистент читает систему, но не меняет её. Агент
  проверяет реальные таблицы/поля перед тем, как писать их в ТЗ; если
  MCP-сервер недоступен, работает без него и помечает объекты как
  «уточнить в системе».

Агент отвечает в чат, а когда нужно изменить документ — вызывает нужный
инструмент раздела. В SSE-стриме каждая правка уходит фронтенду дельтой
(`spec`: порядок разделов + тела только изменившихся) — полный Markdown на
каждый шаг не гоняется; целиком документ отдаётся один раз при открытии
сессии (`GET .../spec`: `spec_markdown` + `sections`). Фронтенд ведёт карту
`id раздела → текст` и собирает превью локально.

История диалога перед каждым вызовом LLM пропускается через бюджеты
(`app/graph/history.py`): старые результаты инструментов заменяются
заглушками, длинные аргументы обрезаются, при переполнении
`HISTORY_MAX_CHARS` старые ходы отбрасываются целиком. Блок релевантных
разделов в промпте тоже ограничен (`SPEC_RELEVANT_MAX_CHARS`). В чекпоинте
история хранится полной — обрезается только то, что уходит модели.

## Авторизация

Доступ ко всем `/api/*` (кроме `/api/health`) закрыт логином через LDAP
(`app/auth.py`). После успешного `/api/auth/login` бэкенд выдаёт JWT-сессию в
httpOnly-cookie; фронтенд показывает экран логина, пока сессии нет.

Поддержаны два режима, выбираются настройками из `.env` (см. `.env.example`):

- прямой bind по `LDAP_USER_DN_TEMPLATE` (OpenLDAP и т.п.);
- search+bind сервисным аккаунтом `LDAP_BIND_DN`/`LDAP_BIND_PASSWORD` — для
  Active Directory, где DN пользователя заранее не известен.

Для локальной разработки без доступа к LDAP-серверу можно поставить
`AUTH_ENABLED=false` — тогда логин не требуется (все запросы идут от
условного пользователя `dev`).

## Сессии

Каждая сессия (thread LangGraph) привязывается к залогиненному пользователю
(`app/sessions.py`) и переживает перезапуск бэкенда — SQLite-файл
`CHECKPOINT_DB_PATH` хранит и чекпоинты LangGraph (переписку, документ ТЗ), и
метаданные сессий (владелец, заголовок, время последней активности). После
каждого хода чекпоинты сессии подрезаются до `CHECKPOINT_KEEP_LAST` последних
— каждый чекпоинт несёт полную копию состояния, без подрезки файл растёт на
мегабайты за ход. Кэш эмбеддингов разделов рабочего документа лежит отдельно
(`SECTION_INDEX_DB_PATH`), чтобы векторы не копировались в каждый чекпоинт.

Заголовок сессии выставляется автоматически по первому сообщению или имени
загруженного файла (можно переименовать вручную). Список сессий пользователя
— `GET /api/sessions`; доступ к чужой сессии по её id закрыт (404). Фронтенд
показывает боковую панель со списком — как история чатов в ChatGPT: можно
переключиться на прошлое ТЗ, начать новое, переименовать или удалить сессию.

## Запуск

### Qdrant (база примеров ТЗ)

```bash
docker compose up -d qdrant     # http://localhost:6333
```

Для разработки без Docker можно указать `QDRANT_URL=:memory:` (встроенный
режим, примеры живут до перезапуска процесса) или локальный путь
(`QDRANT_URL=./data/qdrant` — с сохранением на диск).

### Backend

```bash
cd backend
python -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt
cp .env.example .env   # указать LLM_BASE_URL / LLM_API_KEY / LLM_MODEL
uvicorn app.main:app --reload --port 8000
```

### Frontend

```bash
cd frontend
npm install
npm run dev            # http://localhost:5173, /api проксируется на :8000
```

### Наполнение базы примеров

База пополняется через UI: пилюля «База примеров» в шапке открывает панель,
где можно приложить несколько реальных ТЗ (.docx / .md / .txt) за раз,
посмотреть проиндексированные документы и удалить ненужные. Ключ документа —
имя файла: повторная загрузка с тем же именем заменяет прежние чанки.
База общая для всех пользователей.

## API

| Метод | Путь | Назначение |
| --- | --- | --- |
| POST | `/api/auth/login` | логин по LDAP → httpOnly-cookie сессии |
| POST | `/api/auth/logout` | выход, сброс cookie |
| GET | `/api/auth/me` | текущий пользователь сессии |
| POST | `/api/sessions` | создать сессию (thread LangGraph), владелец — текущий пользователь |
| GET | `/api/sessions` | список сессий текущего пользователя (id, заголовок, время) |
| PATCH | `/api/sessions/{id}` | переименовать сессию |
| DELETE | `/api/sessions/{id}` | удалить сессию и её чекпоинты |
| POST | `/api/sessions/{id}/messages` | сообщение в чат → `{reply, spec_markdown}` |
| POST | `/api/sessions/{id}/messages/stream` | то же, но SSE-стрим: `token`, `status`, `spec` (дельта: `order` + `changed`), `done` (`reply` + `order`), `error` |
| POST | `/api/sessions/{id}/spec/upload` | положить существующее ТЗ в документ (multipart, без LLM) → `{spec_markdown, sections}` |
| GET | `/api/sessions/{id}/spec` | текущий документ: `{spec_markdown, sections}` |
| GET | `/api/sessions/{id}/spec/export` | скачать `.docx`: патч исходного файла, если сессия начиналась с `.docx`, иначе сборка из Markdown |
| GET | `/api/sessions/{id}/messages` | история переписки сессии (для восстановления чата в UI) |
| GET | `/api/examples` | документы базы примеров (имя, ключ, число чанков) |
| POST | `/api/examples/upload` | проиндексировать примеры ТЗ (multipart, несколько файлов) |
| DELETE | `/api/examples?source=...` | удалить документ из базы примеров |

## Деплой в Kubernetes

Всё необходимое лежит в `k8s/` (kustomize): namespace, ConfigMap, Secret,
Deployment'ы backend/frontend, StatefulSet Qdrant с PVC, PVC для
SQLite-сессий (`checkpoints-data`), Ingress.

Схема: Ingress → frontend (nginx: статика + прокси `/api` на сервис
`backend:8000`) → backend → Qdrant и LLM-шлюз.

```bash
# 1. Собрать и запушить образы (docker или podman)
docker build -t <registry>/abap-spec-assist/backend:v0.1.0 backend
docker build -t <registry>/abap-spec-assist/frontend:v0.1.0 frontend
docker push <registry>/abap-spec-assist/backend:v0.1.0
docker push <registry>/abap-spec-assist/frontend:v0.1.0

# 2. Указать свои образы/тег
cd k8s && kustomize edit set image \
  abap-spec-assist/backend=<registry>/abap-spec-assist/backend:v0.1.0 \
  abap-spec-assist/frontend=<registry>/abap-spec-assist/frontend:v0.1.0

# 3. Настроить окружение
#    - k8s/configmap.yaml: LLM_BASE_URL, модели и т.д.
#    - k8s/secret.yaml:    LLM_API_KEY (или создать секрет вручную)
#    - k8s/ingress.yaml:   host

# 4. Применить
kubectl apply -k k8s/
```

База примеров наполняется через UI (панель «База примеров» в шапке) —
отдельного шага деплоя не требует.

Особенности:

- `backend` намеренно в 1 реплику: сессии — SQLite-файл на RWO-томе
  (`checkpoints-data`). Для масштабирования нужен общий чекпоинтер (postgres);
  база примеров живёт в Qdrant и реплик не боится.
- Проба готовности/живости бэкенда — `GET /api/health`; фронтенд стартует и
  до появления бэкенда (nginx резолвит имя сервиса в рантайме, а не на старте).
- Frontend-образ универсален: адрес бэкенда задаётся env `BACKEND_URL`.

## Ограничения / что дальше

- Чекпоинтер — SQLite-файл на одном томе, поэтому `backend` держим в 1 реплику;
  для нескольких реплик нужен `langgraph-checkpoint-postgres`. Рост БД сдержан
  подрезкой до `CHECKPOINT_KEEP_LAST` чекпоинтов на сессию и вынесенным кэшем
  эмбеддингов, но полный список разделов всё ещё дублируется в каждый
  оставленный чекпоинт.
- Разбивка документа на разделы (`app/spec_doc.py`) — эвристики, а не
  парсинг реальной структуры документа: на нетипично оформленных ТЗ без
  заголовков/нумерации возможны неточные границы разделов (`synthetic`).
- Экспорт `.docx` для сессий, начавшихся с загрузки `.docx`, — патч оригинала
  (`app/originals.py`): заменяются только блоки изменённых разделов, поэтому
  шаблон, титульный лист, колонтитулы, картинки и стили нетронутых разделов
  сохраняются. Внутри изменённого раздела картинки/тонкое форматирование
  заменяются сгенерированным текстом. Если карта «раздел → блоки» не
  построилась (аномальный документ без границ абзацев) — полная сборка из
  Markdown (`markdown_to_docx`), которая воспроизводит типовые элементы ТЗ:
  заголовки, жирный/курсив, списки, таблицы.
